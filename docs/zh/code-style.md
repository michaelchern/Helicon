# 代码风格说明

## 目的

这份文档用于给 AI 一个可重复执行的代码格式化和命名基线，方便后续做大规模整理时不再每次重新约定。

## 当前基线

- `include/utils.h` 是当前仓库的参考头文件。
- 新的文件头模板、分组注释样式、双语注释写法，优先向 `include/utils.h` 对齐。
- 在没有更高优先级约束时，Helicon 自有辅助代码优先使用蛇形命名。

## 文件头模板

- 使用统一的 banner 头。
- 保留 `@file`、`@brief`、`@brief.zh`、`@project`、`@author`、`@date`。
- 如果一个文件明显需要说明“英文注释给 AI、中文注释给人”，可以补 `@note.en` 和 `@note.zh`。

## 注释约定

- 英文注释主要服务 AI 检索、推理和代码补全上下文。
- 中文注释主要服务作者自己快速理解职责、用途和边界。
- 对外暴露的函数、结构、类，优先使用 Doxygen 风格双语注释。
- 分组说明优先使用：

```cpp
//-----------------------------------------------------------------------------
// English Section Title
// 中文分组标题
//-----------------------------------------------------------------------------
```

- 不要写低信息量注释，例如“给变量赋值”这种机械描述。

## 命名约定

在明确采用仓库风格的 Helicon 自有代码里：

- 自由函数使用 snake_case
- 辅助方法使用 snake_case
- 参数名使用 snake_case
- 成员变量使用 snake_case

当前不强制修改的对象：

- 已有类型名
- 外部 API 名
- 宏名
- include guard
- 受 shader / ABI / 上游规范约束的字段名和结构名

## 兼容头规则

像 `include/rhiHLSL.h` 这类共享兼容头，优先统一的是：

- 文件头模板
- 分组注释
- 双语说明
- 局部排版

默认不要自动修改：

- 上游镜像结构名
- 规范约束字段名
- 会影响 shader 侧布局或外部接口兼容性的命名

## 大规模格式化规则

- 先按文件或模块分批处理，不要一上来全仓库重命名。
- 遇到兼容层、上游镜像、shader 共享结构时先停下来判断约束。
- 如果用户没有明确要求，不要把“格式化”扩展成“语义重构”。
- 如果风格规则有变化，先更新中文文档，再同步英文 Agent 文档。
