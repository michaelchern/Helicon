# 架构说明

## 当前阶段

Helicon 目前处在探索和装配阶段，还没有最终定版的框架、目录边界或验证流程。

当前更适合的开发方式是：

1. 先由你挑选想保留或想参考的代码，复制进仓库。
2. 再和 AI 讨论规划、边界、命名、目录和取舍。
3. 在几轮删减、调整、拼接和重写之后，把真正稳定的结论写进 Agent 文档。
4. 让后续开发更多依赖沉淀后的上下文，而不是每次重新解释背景。

## 当前约定

- 仓库名和现有代码都和渲染方向相关，但当前实现仍视为候选材料，不等于最终架构。
- 现阶段优先沉淀：保留哪些模块、目录怎么分工、接口边界怎么定、哪些依赖值得继续用。
- 现阶段不要假设：固定图形 API、固定公共 API、固定测试矩阵、固定构建流程。
- 如果某段代码只是临时参考、等待筛选或可能删除，不要在文档里把它写成长期承诺。

## 目录职责

- `include/`：当前保留的公开头文件候选区；接口未定前不要把这里写成稳定 API 承诺。
- `src/`：当前保留的实现和实验代码；允许存在过渡结构和占位目录。
- `examples/`：示例或临时验证入口；是否长期保留要看后续迭代。
- `tests/`：给后续稳定后使用的验证代码预留位；当前可以为空。
- `scripts/`：仍然有意义的可复用脚本入口；不为了“看起来完整”补脚手架。
- `docs/zh/`：给人看的中文源说明。
- `docs/`：给 Agent 用的英文执行上下文。
- `.codex/private/`：本地私有方案、路线、参考仓库和实验记录，不进入 Git。
- `third_party/`：第三方代码预留区；新增依赖前先说明为什么值得留下。

## 文档沉淀原则

- 只记录已经稳定下来的决定、约束和工作方式。
- 方案讨论、临时想法、一次性比较不要长时间留在主文档里。
- 中文先改，再同步英文。
- 如果旧文档已经明显过期，优先删减或重写，不保留历史包袱式说明。

## 构建和测试

当前不要求每次迭代都能构建或测试。

当框架、保留代码和依赖还在变化时，跳过构建或测试是正常的；但需要在变更说明里写明为什么跳过。
只有在某条构建链路或验证流程被明确恢复后，才把它重新写成标准要求。
